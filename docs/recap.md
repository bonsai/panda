## 素直な感想

### 面白いポイント

- **再帰CTEで勝敗判定をSQLだけで書いてるのは純粋にクール**
  - `gomoku_win` は再帰CTEで4方向を走査し、5並びを検出する
  - `WITH RECURSIVE fwd/rev` で最終手から正負両方向に歩き、連続数をカウント
  - 外部コードゼロ、SQLオンリーで閉じてる

- **3ゲームが同じテーブル構造で動くのは設計として気持ちいい**
  - `gomoku_board` と `gomoku_moves` を全ゲームで共用
  - ゲーム切替 = トリガーと勝敗ビューの差し替えだけ
  - スキーマレベルのポリモーフィズム

- **「sqlite3しか要らない」というミニマルさは正義**
  - インストール = `sqlite3 panda.db ".read schema.sql"`
  - 依存ゼロ、設定ゼロ、ランタイムゼロ
  - setup.ps1 ですら「無くても動く」オプショナル

- **オセロの8方向フリップをトリガー8連打で実現したのは無駄に職人芸**
  - 1トリガーに8個のUPDATE、それぞれが1方向のフリップを担当
  - 端点検出は `coalesce(MIN(x)-1, 0)` で賢くガード
  - 保守性より「動くこと」を優先した割り切り

### しっくり来てないポイント

- **「SQLでゲーム」はコンセプトとして尖ってるが、遊ぶためだけなら普通の五目並べの方が100倍快適**
  - SQLクエリを打って遊ぶ、という体験自体がニッチ
  - このプロジェクトの価値は「遊ぶこと」ではなく「SQLでここまでできる」というデモンストレーション
  - プロダクトとしてのUXより、技術デモとしての側面が強い

- **オセロのトリガーは8方向ベタ書きで保守性が低い**
  - もっとSQLiteらしい書き方がある気がする
  - CTEで方向テーブルと結合する方法も検討したが、トリガー内では実現できず
  - 今の8連打は「とりあえず動く」状態。リファクタリング候補

- **勝敗判定が最終手だけ見てるので、「5並びが2本同時にできた」のようなエッジケースで誤検出の可能性**
  - 現実の五目並べではほとんど起こらないが、SQLの正しさとしては気になる
  - 全手を対象にチェックするバージョンも考えられる（ただしパフォーマンスとのトレードオフ）

- **ADRやPRDに対してコードの量がまだ少なすぎる（ドキュメントがコードを喰ってる）**
  - 3ゲームで合計 ~300行のSQLに対し、ドキュメントが ~200行
  - プロトタイピング段階では健全だが、このままドキュメントだけ増えるのは不健康

### 一番気になってること

**「これ、誰が何のために使うんだ？」という問いにまだ答えが出てない。**

SQL学習教材？ ターミナルゲーム集？ SQLiteのトリガー/CTEのショーケース？
PRDには「SQLが好きで無駄なことを真面目にやりたい人」と書いたけど、
それが刺さる人は確実にいる一方、母数は極めて小さい。

### 結論

技術的には良くできてる。でも「作ったこと」自体が目的化していて、
プロダクトとしての体温が低い。

もう一歩進むなら、この仕組みを使って**何か見せられるもの**
（美しい棋譜リプレイ、SQLクエリコンテスト、etc.）が必要。

## PdM視点: 教育コンテンツ化の仮説

### コア価値再定義

「SQLiteの機能をゲームで楽しく学ぶ」が最も刺さるポジション。

| 対象ユーザー | 課題 | Pandaの価値 |
|-------------|------|------------|
| SQL初学者 | 教科書が退屈、実践が欲しい | クエリを打つと画面が動く即時フィードバック |
| 経験者 | CTEやトリガーを書く機会が少ない | 再帰CTE・トリガーを「必要に迫られて」書ける |
| 教育者 | 教材のネタが欲しい | 動くデモ + Issue駆動の課題設計 |

### 教育コンテンツの具体案

1. **チュートリアルモード**
   - 「SELECTで盤面を見よう」→「INSERTで石を打とう」→「CTEで勝敗判定を読もう」
   - 各ステップに解説付き、段階的にSQLスキルが身につく

2. **クエリチャレンジ**
   - 「3手で勝つクエリを書け」「最大の石を返すSELECTを最適化せよ」
   - Issue #2〜#4 を課題として再利用

3. **リプレイ機能の教材化**
   - 棋譜 = 時系列データの典型例
   - `window function` や `LAG` の実践教材に

4. **SQLの限界に挑む**
   - 「SQLだけでAI対戦はどこまで可能か？」
   - CTEで評価関数を書く試み自体が教材価値を持つ

### 教育コンテンツとしての強み

- **環境構築不要**: sqlite3 だけで動く = 学習コストゼロ
- **即時フィードバック**: クエリ結果が盤面として可視化される
- **段階的難易度**: SELECT → INSERT → CTE → トリガー → 最適化
- **拡張性**: 新しいゲームを追加する行為自体がSQL学習になる

### 課題

- 英語/日本語の教材作成コスト
- チュートリアルモードの実装（現状は生sqlite3）
- 学習曲線とゲーム体験のバランス
